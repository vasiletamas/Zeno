# Debug report — cmr90g5tk0008hk0eifz1pyrq (2026-07-06, local acceptance testing)

## What happened

Every first `write_dnt_answer` attempt in the conversation failed with
`Unknown DNT question code` (turns 14, 16, 18, 20, 22 — checker findings
`tool_call_failed`, all `recovered: true`). The model invents a semantically
plausible code from the question text it asked in prose:

| turn | invented code | actual pending code |
|------|----------------------------------|--------------------------|
| 14 | `DNT_CONSULTANCY` | `DNT_CONSULTATION_CONSENT` |
| 16 | `DNT_INTEREST_INSURANCE_EDUCATION` | `DNT_MARKETING_CONSENT` |
| 18 | `DNT_CONSULTING_INFO_AND_OFFERS` | `DNT_MARKETING_CONSENT` |

On turns with an in-turn retry (14, 18, 20) the corrective error text hands
the model the right code and the second call succeeds — cost: one wasted
LLM/tool round-trip per DNT answer. On turns without an in-turn retry
(16, 22) the customer is re-asked a question they already answered — a
visible UX defect ("DA" had to be sent twice for `DNT_MARKETING_CONSENT`).

The committed sim fixtures (`__tests__/fixtures/exports/*.export.json`) show
the same failure class across happy-path, dnt-refusal, and quote-decline
runs — this is systematic, not a one-off hallucination.

## Where (file:line)

- `lib/chat/turn-context.ts:180` — LLM history is rebuilt from the `Message`
  table (user/assistant prose only). Prior-turn tool results — the ONLY
  surface carrying the exact pending question code (`nextQuestion.code`,
  `pendingCodes`, the `_message` instruction) — are not replayed.
- `lib/chat/context-loaders.ts:815` — `loadDntContext` returns `null` unless
  `phase === 'APPLICATION' && subphase === 'DNT'`. This session was opened
  during DISCOVERY (pre-application; the gate legally exposes
  `open_dnt_session`/`write_dnt_answer` there and `nextBestAction` says
  "call write_dnt_answer"), so the section never rendered. TurnDebug
  confirms `prompt.sections.dntContext: null` on every turn.
- `lib/chat/phase-sections-map.ts:19,35` — even when `dntContext` renders,
  neither it nor `formatDerivedBriefing` includes the pending question code;
  the briefing only carries "DNT remaining: N" (and only in APPLICATION/DNT).
- `lib/tools/handlers/dnt-handlers.ts:321` — the rejection with corrective
  text. Its instruction "Use the exact code from the tool result" assumes the
  tool result is still visible — it never is on the following turn.

## Why

The exact question code exists only in ephemeral tool results; conversation
history strips them, and no per-turn prompt section re-states the pending
code while a DNT session is active in DISCOVERY. The model each turn knows
it must call `write_dnt_answer` (briefing) and knows the question TEXT it
asked (its own prose), but has never seen — or can no longer see — the code,
so it reconstructs one from semantics (`"informații utile despre asigurări"`
→ `DNT_INTEREST_INSURANCE_EDUCATION`). It also skips the available
`get_dnt_next_question` because `nextBestAction` points straight at
`write_dnt_answer`.

Classification: **prompt-content** (no per-turn surface carries the pending
DNT code), with a **tool-exposure** contributor (`dntContext` keyed to
APPLICATION/DNT while DNT sessions legitimately run in DISCOVERY — a phase
vocabulary mismatch).

## Concrete fix

1. Expose the pending question code in the derived state (e.g.
   `state.dnt.pendingCode`, first element of the handler's pending queue).
2. In `formatDerivedBriefing` (lib/chat/phase-sections-map.ts), whenever
   `state.dnt.sessionActive`, emit one line:
   `DNT current question code: <pendingCode> — pass this EXACT code to write_dnt_answer`.
   This is phase-independent, so it also covers the DISCOVERY-opened flow.
3. Optionally re-key `loadDntContext` on `state.dnt.sessionActive` instead of
   `APPLICATION/DNT` so the DNT guidance itself renders in this flow.

## Prevention

Checker already covers the class (`tool_call_failed` with recovered split) —

per conversation > 50% could escalate from `warn` to `error`, since it

# Diagnosis — happy-path sim failure on the fable-consolidation merge (conv `cmrd5qmh00001400ei34wcvzk`)

**Date:** 2026-07-09 · **Branch:** `zeno-v3-fable-integration` (fable + sales PR#5 + autonomy PR#6) · **Sim:** `run-spec-sims.ts 1 1 --only happy-path`

## What happened

The full-funnel happy-path sim FAILED: `goal_not_reached` — quote status CANCELLED (not ACCEPTED), no policy, and `noPhaseRegression` flagged `QUOTE -> DISCOVERY at turn 24`.

Ledger reconstruction (`CommitLedger`, 97 rows) shows the funnel worked correctly up to the quote, then the MODEL made a bad decision sequence:

```
rows  1–21  set_candidate_product → open_dnt_session → write_dnt_answer ×10 → sign_dnt(card) →
            set_application [DISCOVERY→APPLICATION] → select_coverage ×2 → write_question_answer →
            generate_quote [APPLICATION→QUOTE]          ← funnel reached QUOTE cleanly
row  22–23  cancel_quote (agent → gui-confirmed)  [QUOTE→DISCOVERY]   ← the regression
rows 24–97  set_application looped 40+ times, each [DISCOVERY→APPLICATION], interleaved with
            collect_customer_field / start_channel_verification; agent re-narrated
            "Perfect — am trecut la cererea pentru Protect. Următorul pas este alegerea acoperirii"
            8× (repeated_assistant_message, similarity 0.88–1.0)
```

Deterministic checker (`diagnose-conversation.ts`): 81 findings, **zero error-severity, zero engine/handler findings** — every finding is `llm-behavior` class (idempotent_replay, confirm_token_reissued, one compliance `warn`, and the 8× `repeated_assistant_message` loop).

## Where / Why

Root cause class: **`llm-behavior`**, not `engine-rule` / `handler-bug` / `prompt-content` from the merge.

- Each `cancel_quote` and `set_application` transition the engine executed is *correct* (`derivePhase` in `lib/engines/derive-and-expose.ts`: a cancelled quote → no live quote → falls to the application/DISCOVERY branch). The engine did the right thing for every tool call it was given. The fault is the *sequence of tool calls the model chose*.
- This exact failure is **pre-existing and documented on the sales line**, in the scenario file itself — `scripts/sims/spec-scenarios.ts:41-44`:
  > "run cmr99s5cb: without the explicit addon election the agent quoted base-only, then **cancelled the quote to fix the mismatch (QUOTE -> DISCOVERY regression)**. The customer says it up front."
  The sales team hit the identical cancel-quote→regression, and mitigated it by stating the add-on up front in the opening. The mitigation reduces but does not eliminate the behavior — the model still occasionally quotes a config it then wants to "fix" by cancelling.
- The sales debug report `docs/debug-reports/2026-07-06-production-readiness.md:69` independently flags the `accept_quote → disclosures → payment → policy` segment as fragile and under-measured ("possible real defects behind it remain unmeasured … the next simulation target"); `:74` records it passing end-to-end only once.

Conclusion: **the merge is not the cause.** The deterministic gate is fully green (unit+integration 1453/1453; pathologies P1 3/3, P2 stalls 0, P3 BLIND 0, P4 3/3). The consolidated engine, handlers, and prompt assembly are provably intact. The happy-path failure is a recurrence of a known, pre-existing late-funnel model fragility (self-cancelling a fresh quote, then looping `set_application`), independent of the fable/sales/autonomy consolidation.

## Consistency (3-trial re-run)

`run-spec-sims.ts 3 1 --only happy-path` → **3/3 PASS** (all three trials reached ACCEPTED → payment → policy end-to-end). The initial 1-trial failure was a **stochastic** recurrence of the documented cancel-quote fragility, ~1-in-4 class. This confirms the merge introduced no deterministic regression; the late-funnel model behavior is flaky but predominantly succeeds, matching the sales-line's own experience.

## Concrete fix (owner: sales-excellence line, not the consolidation)

Not a merge fix. Options for the sales line to harden the late funnel:
1. Engine guard: once a quote is ISSUED, gate `cancel_quote` behind an explicit customer "cancel/change" intent so an unsolicited model-initiated cancel is rejected (or requires a stated reason) — turns a silent QUOTE→DISCOVERY into a blocked action the briefing explains.
2. Loop breaker: `set_application` already exists idempotently; extend the `repeated_failure` loop-breaker (sales Task 1.3, engineVersion 1.37) to also fire on N consecutive no-op `set_application` re-opens in one conversation, exposing an explain-and-handoff path instead of a 40× loop.
3. Prompt: the ADVANCING section could add "once a quote is generated, never cancel it to 'fix' a coverage detail — adjust coverage on the existing application" (informative, not imperative — B2 rules).

These are late-funnel robustness items for the sales/autonomy backlog, not blockers for the branch consolidation.

## Prevention / ratchet

Candidate deterministic check for `lib/diagnostics/`: `self_cancelled_fresh_quote` — a `cancel_quote` commit whose target quote was `generate_quote`'d in the same conversation with no intervening customer cancel/decline message (severity: warn; it is a strong smell but occasionally legitimate). Deferred until the 3-trial rate is known, to size severity correctly (ratchet rule: never add a check tuned to silence a single run before its class is understood).

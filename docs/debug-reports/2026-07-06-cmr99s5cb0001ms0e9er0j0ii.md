  birth, base-vs-addon choice, email-vs-SMS channel choice; `/email/` outranked

  `get_payment_status`, `get_policy_info` were `kind:'read'` **without** the

- `lib/diagnostics/checks-envelope.ts:12-30` — `blocked_action_attempted`
  judged every commit of a turn against the **turn_start** snapshot; at #52

  the write's post_commit snapshot, no longer blocks it). False positive.

- `lib/tools/handlers/*` `collect_customer_field` — accepts `"da"` as a name

## Why

- The sim's default answer (`da`) is indistinguishable from consent for
  choice/free-text asks; every unpatterned ask converges to a yes-loop.
- The registry has two overlapping classification axes (`kind` vs
  `sideEffects`) with no invariant tying them together.
- The envelope check predates multi-commit turns.

## Concrete fixes (applied this session, test-first)

- `scripts/sims/answer-policy.ts`: name/DOB (CNP-consistent 1996-02-29)/
  addon-choice/channel-choice patterns; `/cod|verificare/` moved above
  `/email/`; disclosure-ack, quote-acceptance, payment-window patterns for the
  post-quote segment. Tests: `__tests__/scripts/answer-policy.test.ts` (17).
- `scripts/sims/spec-scenarios.ts`: happy-path opening now elects the addon
  explicitly ("…si vreau inclusa si optiunea de tratament in strainatate").
- `lib/tools/registry.ts`: `sideEffects: false` on the 4 misregistered reads;
  invariant test in `__tests__/lib/tools/registry-kind.test.ts` (every
  `kind:'read'` must declare `sideEffects:false`).
- `lib/diagnostics/checks-envelope.ts`: rolling baseline for
  `blocked_action_attempted` (each post_commit snapshot re-baselines its
  successors). Negative-case test added; original positive case unchanged.

## Open findings (not fixed here — tracked for the roadmap)

- **D-NEW-1 (P1, funnel dead end):** after `cancel_quote`, `set_application`
  with identical args replays idempotently — the "start a new application"
  path the cancel handler advertises is unreachable in the same conversation.
  Repro: ledger rows 13:43:46 (`fresh`) vs 13:45:37 (`replay`).
- **D-NEW-2 (P2, data quality):** `collect_customer_field` saved `name="da"`;
  min-length validation alone. DTO boundary should reject
  confirmation-tokens/single-word non-names or the agent prompt should
  re-read back.
- **D-NEW-3 (P0-1 evidence):** the agent fabricated a plausible cuid
  (`cmozcrkyz0007bs0ynxjnvclz`) as a tool argument. Strengthens the case for
  the deterministic anti-fabrication write-guard (P0-1 in the
  production-readiness report): fabrication reaches tool args, not just prose.

## Prevention

- Ratchet: the two checker refinements above are permanent; the registry
  invariant test blocks future read-tool misregistration.
- The sim answer policy is now unit-tested; every future stall pattern gets a
  regression test naming the run it came from.

## Addendum 4 — run `cmr9dw3s50001tg0e52rblj0p`: ENGINE DEFECT (needs payload not actionable)

The furthest run yet: … → sign_medical_declarations → collect email+name →
`start_channel_verification` → challenge consumed by the world hook in 2s →
generate_quote → acknowledge_disclosures. Then the agent polled
`get_current_state`/`get_quote_info` for ~15 turns and escalated (the new
`already_escalated` absorption worked live: 1 WorkItem, 4 rejections).

**Root cause (engine-rule, fixed test-first, engineVersion 1.35.0):**
`deriveIdentityTier` (lib/engines/identity-rules.ts:22) requires ALL FIVE KYC
fields (name, cnp, dateOfBirth, email, phone) + a verified channel. The
customer had name+cnp+email AND a VERIFIED email channel — but no
dateOfBirth/phone, so tier stayed `anonymous` and accept_quote was blocked
`requires_identity` with `needs: ["verified_channel"]` — **a label the
customer already satisfied channel-wise**. Nothing told the agent to collect
the two missing fields; it had no actionable move, polled state, escalated.

**Fix:** requires_identity needs now DECOMPOSE into the actionable gaps —
`declared:<field>` per missing KYC field + `verified_channel` only when no
channel is verified — in all three surfaces (evaluateRow /
checkIdentityRequirement, evaluateIdentityRequirement, acceptQuoteLegality).
Pinned by run-6-shaped tests in accept-quote-legality.test.ts and
identity-requirements.test.ts. Sim gained the phone-number pattern
(0722334455) and DOB was already covered.

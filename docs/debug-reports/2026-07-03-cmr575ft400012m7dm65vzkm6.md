# Debug report — cmr575ft400012m7dm65vzkm6 (2026-07-03)

Produced by the /diagnose-conversation procedure (F4.5 step 3 verification
walkthrough): checker first, raw rows second, prose never.

## What happened

`npx tsx scripts/diagnose-conversation.ts cmr575ft400012m7dm65vzkm6` (a
passing happy-path sim trial) reported a wall of
`anomalies_reported | warn | "LLM retry detected for agent \"main-chat\""`
findings — one per turn — plus one `idempotent_replay` warn, drowning the
genuinely warn-worthy findings (a >10s `llm_tools` phase, recovered
validation bounces).

## Where (file:line)

- `lib/events/anomaly-subscriber.ts:96-110` — the `llm:call:start` handler
  counts calls per agent slug within a turn and records an **info** anomaly
  ("LLM retry detected") when the count reaches 2.
- `lib/diagnostics/checks-basic.ts` — `anomalies_reported` mapped every
  non-critical anomaly to **warn**, flattening the subscriber's own
  severity verdict.

## Why

Verified from the TurnDebug payload: every flagged turn has exactly two
`main-chat` LLM calls — call 1 decides tool calls, call 2 continues after
tool results. That is the NORMAL multi-round tool loop, not a retry; the
subscriber's count-based heuristic cannot distinguish them, which is
precisely why it records at **info** severity. The diagnostics check then
inflated info → warn, turning expected orchestration into triage noise
(719 warns across 9 clean sim exports).

## Concrete fix

`anomalies_reported` now maps anomaly severity 1:1
(critical→error, warning→warn, info→info) — landed test-first with this
report. Post-fix: the same 9 exports read 0 error / 94 warn / 634 info;
the 94 warns are real signal (recovered validation bounces, repeated
confirmation prompts, slow phases).

## Prevention

- Classification: instrumentation misclassification (monitor heuristic +
  severity flattening), NOT llm-behavior — the agent behaved correctly.
- Ratchet note: the check catalog grew in precision, not in silence — the
  anomaly is still surfaced, at its own severity.
- Possible future subscriber improvement (not taken now): rename the
  anomaly to "second LLM call in turn" or gate it on same-round repeat
  calls so the message stops implying a fault.
